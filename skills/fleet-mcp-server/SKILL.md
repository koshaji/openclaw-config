# fleet-mcp-server

> **Status:** Phase 4 — Implemented
> **Tracking:** GAP_CLOSING_PLAN.md, Gap 1 (Zero-SSH Fleet Operations) + Gap 4 (NL Fleet Control)
> **Reference:** [MCP Protocol](https://modelcontextprotocol.io)

## Overview

`fleet-mcp-server` is a UV-based MCP (Model Context Protocol) server that exposes
fleet operations as typed tools to AI agents. It bridges the gap between natural
language fleet commands and the low-level inbox/outbox protocol used by `fleet-agent`.

The fleet commander workflow (`workflows/fleet-commander/AGENT.md`) calls this server
to execute fleet operations. Human operators can also use it directly via any
MCP-compatible client.

## Architecture

```
Atlas4 (NL command)
    ↓
Fleet Commander Workflow (intent → tool mapping)
    ↓
fleet-mcp-server (MCP tools via stdio/SSE)
    ↓
~/.openclaw/fleet-inbox/cmd-<uuid>.json  (HMAC-signed command files)
    ↓
fleet-agent (on each remote machine, polling inbox)
    ↓
~/.openclaw/fleet-outbox/result-<uuid>.json (results)
    ↑
fleet-mcp-server (polls for results, returns to caller)
```

## MCP Tools

| Tool | Description | RBAC Required |
|------|-------------|---------------|
| `fleet_status` | List all machines with health | read |
| `fleet_health_check` | Run health check on machines | read |
| `fleet_restart` | Restart gateway on machines | write |
| `fleet_update` | Update component on machines | write |
| `fleet_config_push` | Push config to machines | admin |
| `fleet_logs` | Get recent logs from a machine | read |

## Prerequisites

- Python 3.11+ (managed by UV)
- `mcp` package (PyPI: `mcp`) — UV handles installation
- `~/.openclaw/fleet/inventory.json` — fleet machine list
- `~/.openclaw/fleet/fleet-hmac-key` — shared HMAC key (generated by fleet-agent setup)
- `~/.openclaw/fleet/rbac.json` — RBAC configuration

## Installation

```bash
# Make executable
chmod +x skills/fleet-mcp-server/fleet-mcp-server

# Verify it runs
./skills/fleet-mcp-server/fleet-mcp-server --help
```

## Usage

### stdio mode (for MCP client integration)

```bash
./skills/fleet-mcp-server/fleet-mcp-server
```

Configure your MCP client (e.g. OpenClaw's `mcp.json`) to use stdio transport:

```json
{
  "mcpServers": {
    "fleet": {
      "command": "/path/to/openclaw-config/skills/fleet-mcp-server/fleet-mcp-server",
      "args": [],
      "env": {}
    }
  }
}
```

### SSE mode (for web/HTTP clients)

```bash
./skills/fleet-mcp-server/fleet-mcp-server --transport sse --port 8766
```

The SSE endpoint will be available at `http://localhost:8766/sse`.

## Inventory File

The server reads fleet machine definitions from `~/.openclaw/fleet/inventory.json`:

```json
{
  "machines": [
    {
      "name": "mac-mini-01",
      "host": "mac-mini-01.tailnet",
      "port": 18789,
      "role": "primary",
      "agents": ["atlas4", "forge4", "nova4"],
      "inbox": "/Users/admin/.openclaw/fleet-inbox",
      "outbox": "/Users/admin/.openclaw/fleet-outbox"
    }
  ],
  "updated_at": "2026-03-29T00:00:00Z"
}
```

For remote machines, commands are delivered via the shared inbox/outbox directories
(typically mounted via Tailscale + NFS, or synced via git). For local machines
(same host), the inbox/outbox paths are local.

## RBAC Configuration

`~/.openclaw/fleet/rbac.json` maps agent identities to permitted operations:

```json
{
  "roles": {
    "atlas4": ["read", "write"],
    "forge4": ["read"],
    "nova4": ["read"],
    "vault4": ["read", "write", "admin"]
  },
  "operations": {
    "read":  ["fleet_status", "fleet_health_check", "fleet_logs"],
    "write": ["fleet_restart", "fleet_update"],
    "admin": ["fleet_config_push", "fleet_update"]
  }
}
```

The server reads the caller's identity from the `X-Agent-Identity` environment variable
or the `OPENCLAW_AGENT_ID` environment variable.

## Audit Log

All operations are appended to `~/.openclaw/audit/fleet-mcp-server.jsonl`:

```json
{
  "ts": "2026-03-29T00:00:00Z",
  "agent": "atlas4",
  "tool": "fleet_restart",
  "machines": ["mac-mini-01"],
  "args": {"graceful": true},
  "result": "ok",
  "command_ids": ["cmd-abc123"]
}
```

## Transport Modes

| Mode | Use Case | Notes |
|------|----------|-------|
| stdio | MCP client integration (OpenClaw, Claude Desktop) | Default |
| SSE | Web frontends, HTTP clients, multi-client scenarios | `--transport sse` |

## Integration with Fleet Commander

The fleet commander workflow calls fleet-mcp-server tools in response to natural
language commands. See `workflows/fleet-commander/AGENT.md` for the routing logic.

Example flow:
```
Human: "check if all agents are healthy"
→ fleet-commander routes to fleet_health_check(machines=None)
→ fleet-mcp-server sends health_check command to all machines
→ fleet-agent responds with health reports
→ fleet-mcp-server aggregates and returns results
→ fleet-commander summarizes in natural language
```

## Troubleshooting

| Issue | Cause | Fix |
|-------|-------|-----|
| `inventory.json not found` | Missing fleet setup | Create `~/.openclaw/fleet/inventory.json` (see template) |
| `RBAC denied` | Agent lacks permission | Add agent to `rbac.json` with required role |
| `Command timeout` | fleet-agent not running on target | Check `fleet-agent status` on target machine |
| `HMAC key missing` | HMAC key not configured | Run `fleet-agent setup` to generate shared key |
